# 标书拆解 · 结构化分析指引

本文件供分析阶段使用。目标：把「抽取出的招标文件 Markdown」转化为 `report-schema.json` 形状的报告 JSON。
**一切以招标文件原文为准**，严禁编造。

---

## 0. 总原则（红线）
- **可溯源**：每条事实性内容都要能指回原文位置，写进该块的 `source`（章节名 / 条款号 / 前附表项）。`source` 缺失即视为不可靠。
- **缺失标注**：招标文件没写清楚的，填「招标文件未提及」，不要脑补、不要照搬其他项目。
- **原文优先**：废标条款、★/▲ 条款、评分公式、付款/违约/分包等关键条目，优先**摘录原文措辞**，再补一句白话解释。
- **图片证据**：抽取文本中出现的 `[IMAGE]` 段落，说明「本节含图片证据（证书/截图/签章/合同），文本无法读取，需核对原件」，并在报告中以 callout 提示核对。
- **章节齐全**：`sections` 必须正好 10 个，id/顺序固定：info / time / reject / qual / score / demand / files / tips / strategy / pit。

---

## 1. meta（报告头）
从「招标/比选邀请、供应商/投标人须知前附表、第一章」提取：
- project_name / project_no / purchase_method（公开招标·邀请招标·竞争性磋商·比选·询价）
- purchaser / agencey（采购人 / 代理机构）
- budget（预算或最高限价；注明超限即无效的，red 标色）
- deadline（响应/投标文件递交截止）、validity（投标/应选有效期）
- agency_fee（代理服务费及承担方）
- hero_tag（如「招标文件 · 拆解分析」）、hero_sub（项目编号 / 采购方式 / 采购人 一行）
- facts_extra：hero 下方 3–5 张关键指标卡（如预算、截止、有效期、代理费），tone 用于标红/标绿。

## 2. sections（逐章）
### ① info 基本信息
facts 网格：项目名称、编号、采购人、代理机构、采购方式、预算/限价、标段、保证金、履约保证金、文件份数/装订、付款方式等。
hint：「来自『招标邀请』与『供应商/投标人须知前附表』」。
callout（如发现前后不一致，如编号矛盾，amber 提示；联系方式 blue 提示）。

### ② time 时间节点
timeline 块：文件获取→递交/开启→有效期→交货/服务期→付款/质保 等关键时点（date/ev/note）。
callout green：用「倒推法」给出定稿、装订、送达建议（基于招标文件给定的开标/递交时间）。

### ③ reject 废标/无效条款（★ 实质性，最高优先级）
text 先说明依据章节。
table：逐条列出「否决/无效情形」（# / 情形 / 类型 pill：形式·资格·信用·报价·签章·关联·兜底）。
callout warn：归类（形式性/资格性/信用类）。
callout amber：关键时点提示（如信用查询时点、保证金免交但仍须复核）。
callout blue：避废标要点清单（汇总可直接照做）。

### ④ qual 资格条件
table：# / 资格要求 / 验证材料（pill：声明函·承诺函·自查截图·单独投标）。
callout green：利好项（如免保证金、免履约保证金，但代理费计入成本）。

### ⑤ score 评分办法
text：评审方法说明（满足实质性后按分高低；同分处理）。
scorebars：各评分项占比（name/pct/pts）。
按招标文件原样摘录各评分项明细（报价公式、技术参数分档、▲ 条款证明分、方案/售后/经验等），用 table / callout / list 表达。
callout green：提醒「未描述即 0 分，宁多勿缺」。

### ⑥ demand 采购需求与技术参数
facts：预算、交货期、质保期、付款、设备/货物项数、服务项、转包分包（禁止则 red）。
text：项目定位/建设目标（摘录或概括原文）。
table：固定资产设备/货物清单（# / 名称 / 单位 / 数量 / 关键技术参数摘要 / ▲ 标记）。参数摘要**来自原文**，不编造指标。
list：配套课程/服务要求（若有）。
callout warn：▲ 重要参数佐证硬性要求（须附检测报告/彩页/截图并标记、盖章）。
callout blue：踏勘/测量提示（若文件要求）。
> 若本文件有「技术参数响应证明材料」的明确要求，见规则七（⑧ 中汇总表或此处单列）。

### ⑦ files 响应文件组成
table：响应文件须包含的条目（# / 组成部分 / 对应附件格式）。
callout amber：装订与签署要求（正本/副本/电子档/光盘U盘）。
callout blue：密封标记字样要求。

### ⑧ tips 解读技巧（7 大）
table：# / 技巧（一~七）/ 在本文件中的落点（source 到具体章节）。
- 一 先看须知前附表；二 圈出★/▲；三 看清报价须知；四 研究评分标准；五 **检查商务条款**（付款周期/履约保证金/违约责任/分包——注意已更名为商务条款）；六 看废标/无效条款；七 **检查技术参数响应证明材料**。
- 技巧五落点：摘录商务/合同条款中的付款、违约、分包等原文。
- 技巧七落点：若文件有技术参数响应证明材料要求，指向 `tech_materials` 汇总表，并说明其用途（团队分工+交叉核对，避免遗漏扣分）。

### 技术参数响应证明材料汇总表（规则七 · 按需）
仅当招标文件对技术参数响应有**明确证明材料要求**时填写 `tech_materials`，否则置 `null`。
每行：clause（参数条款，含 ★/▲）/ material（检测报告/产品彩页/功能截图/厂商授权函等）/ source（章节/条款号）。
在 ⑥ 或 ⑧ 之后以独立 callout/table 呈现，并在 source_note 中说明其来源。

### ⑨ strategy 得分策略
table：评分项 / 分值 / 策略建议（按「必争满分 / 争高分 / 难拿」三类给可执行建议，均 source 到评分办法）。
callout green：给出竞争力基线目标分（参考手册「低于80分谨慎参与」原则）。

### ⑩ pit 避坑清单
list：开标前逐项核对项，pill 分类（废标/得分/材料/风险/成本/现场）。
每条都来自前面章节的硬性要求，不新增无依据项。

---

## 3. source_note（页脚）
一句话声明：数据来源《……招标文件》（项目编号，日期）；本报告依据《标书拆解规则手册》（9 步 + 7 大技巧）自动拆解，供投标决策参考；具体以招标文件原件及代理书面澄清为准。附生成时间。

## 4. 自检清单（输出 JSON 前过一遍）
- [ ] sections 正好 10 个，id/顺序正确
- [ ] 每条事实都有 source（或显式「招标文件未提及」）
- [ ] 废标/★/▲/评分公式 为原文摘录
- [ ] `[IMAGE]` 处均有「需核对原件」提示
- [ ] 有明确技术参数证明材料要求时，tech_materials 非空且每行有 source
- [ ] 无任何「套用其他项目」或主观臆测内容
