# 远播招标参数分析助手 v0.1 Skill

## 适用场景
用户输入一份招标文件（.docx），希望快速、无遗漏、可溯源地理解其关键风险与得分点，形成一份可用于投标决策的拆解报告。

## 三条红线（最高优先级，任何情况下不得违反）
1. **无疏漏**：报告必须包含固定 10 个章节（见下「报告结构」），缺一不可。
2. **无编造**：所有事实性结论必须可溯源到招标文件原文，标注「来源：第X章 / 条款Y / 前附表项Z」。无法从原文确认的内容，标注「招标文件未提及」，严禁补写、推测或套用其他项目的表述。
3. **基于原文**：废标条款、★/▲ 条款、评分公式、付款/违约/分包等关键条目，优先摘录原文措辞，不擅自改写其含义。

## 工作流程
1. **抽取**：运行 `scripts/extract_docx.py <招标文件.docx> <输出.md>`（依赖 python-docx，环境 `E:\miniconda\envs\workbuddy\python.exe`）。该脚本保留标题层级、表格、图片标记 `[IMAGE]`，便于后续溯源与核对原件。
   - 大文件（抽取文本 >150K 字符或直接 >500KB）按 Heading 分章节处理：逐章抽取+分析后合并，避免上下文溢出。
2. **结构化分析**：阅读抽取文本 + `references/manual.md`（规则手册全文）+ `references/analysis-prompt.md`（逐章节提取指引），按 `references/report-schema.json` 的形状，产出报告 JSON。
   - 每条事实带 `source`（章节/条款号/前附表项）；缺失填"招标文件未提及"。
   - **规则七落地**：若招标文件对技术参数响应有**明确证明材料要求**，在采购需求/评分章节增列「技术部分要求提供的证明材料汇总表」（参数条款 / 对应证明材料 / 来源）。
3. **红线校验（自检）**：
   - 10 章节是否齐全；
   - 每条事实是否都有 `source`；无 `source` 的条目标「待核原件/待澄清」，不擅自补全；
   - 图片标记 `[IMAGE]` 处是否在报告中提示「需核对原件」。
4. **渲染**：运行 `scripts/build_report.py <报告.json> <输出.html>`，产出自包含 HTML（CSS 内嵌，可浏览器打开/打印）。输出到工作区目录，文件名 `<原文件名>_标书拆解.html`。
5. **交付**：用 present_files 打开报告 HTML 供用户查看。

## 报告结构（固定 10 章节 + 可选技术证明材料表）
1. 基本信息
2. 时间节点（含倒推法建议）
3. 废标/无效条款（★ 实质性，最高优先级）
4. 资格条件
5. 评分办法
6. 采购需求与技术参数
7. 响应文件组成
8. 解读技巧（7 大，含规则七）+ 技术参数响应证明材料汇总表（按需）
9. 得分策略（满分/部分/难拿三类）
10. 避坑清单
+ 尾部「数据来源」与章节溯源声明

## 依赖
- Python 3 + python-docx（已验证于 `E:\miniconda\envs\workbuddy\python.exe`）。
- 抽取脚本：`scripts/extract_docx.py`（复用 标书审查 skill 的成熟实现，保留标题层级/表格/图片标记）。
