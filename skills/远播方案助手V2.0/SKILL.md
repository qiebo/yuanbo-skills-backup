# 远播方案助手 V2.0

## 1. 目标

本 Skill 的目标不是生成“像方案的文字”，而是生产可评审、可核验、可修改、可直接交付的正式方案。

交付物必须同时满足：

- **需求闭环**：用户明确提出的要求全部有落位、有回应、有验收方式。
- **事实可追溯**：政策、学校事实、价格、参数、赛事、合作关系等均可追溯到证据。
- **设计有专属性**：顶层设计基于学校真实基因和项目真实约束，不靠替换校名复用。
- **方案可实施**：课程、设备、空间、人员、进度、预算和验收相互匹配。
- **文档可交付**：Word 结构完整、表格不丢失、目录可用、版式经过渲染检查。

## 2. 强制原则

### 2.1 禁止静默补全

下列信息没有可靠依据时，不得写成确定事实：

- 学校现有基础、师资数量、课程数量、合作高校、获奖情况；
- 设备型号、技术参数、数量、单价、折扣、税率；
- 政策文件名称、文号、发布时间、适用范围；
- 赛事是否在办、是否在当前名单、面向学段；
- “首个、首创、领先、唯一、填补空白、示范标杆”等比较性结论。

处理方式只有三种：

1. 补充调研并登记证据；
2. 改写为建设目标、预期成效或拟议方案；
3. 登记为待确认事项，不进入无标注的正式终稿。

### 2.2 单一内容源

- 正文唯一源文件为 `07_方案终稿.md`。
- 预算唯一源文件为 `data/预算明细.csv`。
- 事实唯一登记入口为 `data/事实证据台账.csv`。
- 用户需求唯一登记入口为 `data/需求响应矩阵.csv`。
- Word 是渲染后的交付物，不作为主要编辑源。

禁止同时在脚本、Markdown 和 Word 中维护三套正文或预算。

### 2.3 失败必须显式暴露

以下情况必须停止交付并输出错误清单，不得回退到示例数据：

- 找不到终稿文件；
- 存在未关闭的高优先级待确认事项；
- 预算不平；
- 高风险事实无证据；
- 用户强制需求未覆盖；
- Word 构建工具不可用；
- 渲染失败或发现明显页面缺陷。

## 3. 文档类型识别

先判断主文类，再选择对应的内容骨架与评审角色。不得把所有方案套入同一六章模板。

| 类型 | 主要用途 | 顶层设计强度 | 核心关注点 |
|---|---|---:|---|
| A 自由建设方案 | 学校特色空间、实验室、科创中心 | 高 | 学校基因、建设逻辑、课程与实施闭环 |
| B 标准申报书 | 政府/教育部门固定字段申报 | 中 | 字段完整、政策匹配、量化指标、合规 |
| C 预算概算/配置清单 | 立项预算、采购前测算 | 低 | 分类、依据、计算、口径、必要性 |
| D 采购技术方案/合作协议 | 设备、平台、服务采购 | 低 | 技术响应、交付边界、验收、风险 |
| E 汇报稿/逐字稿 | 项目汇报、领导汇报、路演 | 高 | 战略判断、表达节奏、事实安全、口径一致 |
| F 课程建设方案 | 课程体系、教师培训、学生手册 | 中 | 目标、学段、课时、任务、评价、资源匹配 |

文类规则见 `references/document_type_rules.md`。

## 5. 工作流

### 阶段 0：执行计划

读取用户材料后，先在 `00_执行计划.md` 中确定：

- 文档类型与交付目标；
- 已有资料、缺失资料和可接受假设；
- 需要调研的事实；
- 预计产物；
- 适用的校验器和评审角色；
- 不确定性与风险。

信息已经足够时直接继续执行，不为形式而反复询问。影响方案方向、预算或真实性的关键缺口才列为决策点。

### 阶段 1：需求建模

将用户要求、会议纪要、模板字段和附件要求拆入 `data/需求响应矩阵.csv`。

每条需求至少包含：

- 需求编号；
- 原始表述；
- 类型（强制/偏好/约束/禁止项）；
- 方案响应；
- 落位章节；
- 验收方式；
- 状态。

强制需求覆盖率必须为 100%。

### 阶段 2：调研与证据登记

调研不是“搜集可用句子”，而是建立事实证据台账。

所有准备写入方案的外部事实登记到 `data/事实证据台账.csv`。字段和用法见 `references/evidence_rules.md`。

事实类型必须区分：

- `verified_fact`：已核实事实；
- `user_provided`：用户提供、未独立外部核实；
- `inference`：基于证据的推断；
- `proposal`：本方案拟议内容；
- `target`：建设目标；
- `expected_outcome`：预期成效；
- `pending`：待确认。

政策与赛事属于时效性信息，必须记录核验日期、适用期限和官方来源。禁止维护“永久有效”的静态赛事状态表。

### 阶段 3：顶层设计决策

先建立学校基因卡，再决定是否需要概念词和结构隐喻。

学校基因至少包括：

- 校训精神及可观察行为转译；
- 校史文脉及与项目主题的真实关联；
- 现有优势及其证据；
- 未来方向及其官方依据；
- 项目缺口与约束。

顶层设计不是所有文档的强制装饰。按以下条件使用：

- 学校级建设方案、整体空间方案、对外汇报：可使用较强概念设计；
- 标准申报书：适度使用，优先服从字段；
- 预算、采购技术响应：弱化概念，强化分类和可验收性。

“首个、领先、示范”等表述必须作为高风险主张登记证据；无法证明时自动降级为“拟形成、重点建设、对标、探索”。

详细方法见 `references/top_design_playbook.md`，其中案例只能作为结构示例，不得作为当前事实直接复用。

### 阶段 4：方案架构与映射

在大纲前完成三张关键矩阵：

1. **需求响应矩阵**：防止漏项；
2. **课程—设备—空间映射**：防止设备无课程、课程无资源；
3. **实施验收矩阵**：防止只写建设内容、不写责任与验收。

课程映射至少包含：能力目标、课程模块、实践任务、学段、课时、设备、空间、班额、教师要求、评价产物。

实施验收至少包含：阶段、责任主体、前置条件、任务、成果、验收标准、时间、依赖和风险。

### 阶段 5：正文撰写

正文要求：

- 论证性内容使用完整自然段；
- 任务、指标、预算、进度、责任、参数优先使用表格或结构化条目；
- 不机械禁止列表，也不使用低信息密度的“首先、其次、最后”填充篇幅；
- 事实、目标、方案和预期成效使用不同语气；
- 每个结论的力度不得超过证据；
- 每章必须回应需求矩阵中的具体需求；
- 校训和学校基因必须转译为建设行为，不得只在结尾贴标签；
- 避免万能形容词、同义反复和无法验收的宏大表述。

写作细则见 `references/writing_guide.md`。

### 阶段 6：确定性校验

先运行硬校验，再进行专家评审：

```bash
python scripts/validate_project.py <项目目录>
```

至少检查：

- 强制需求覆盖率；
- 未关闭待确认事项；
- 高风险事实证据覆盖；
- 预算数量×单价、小计、分类小计和总计；
- 元/万元、含税/未税等口径；
- 课程、设备、空间映射缺口；
- 正文中的待确认标记和高风险词；
- 赛事是否有当期核验记录；
- 关键数字在正文和数据表中的冲突。

校验失败时不得进入交付阶段。

### 阶段 7：自适应专家评审

评审角色按文类选择，不固定为同一组三人：

- 建设方案：教育方案、课程实施、工程/设备、政策合规；
- 课程方案：课程、学科教学、评价、实施资源；
- 预算/配置：采购、财务、技术适配、合规；
- 采购技术方案：技术、交付、运维、安全；
- 汇报稿：战略表达、事实核验、决策者视角。

先独立评审，再汇总。评审使用 0—4 级行为锚定标准，不使用缺乏定义的泛化 10 分制。

一票否决项：

- 虚构事实或来源；
- 预算错误；
- 政策/赛事状态错误；
- 未经授权的合作关系；
- 关键需求遗漏；
- 无法实施或无法验收的关键建设内容；
- Word 核心内容缺失。

量表见 `references/review_rubric.md`。

### 阶段 8：生成 Word

推荐命令：

```bash
python scripts/build_docx.py <项目目录>
```

构建器必须：

- 读取 `project.yaml` 和 `07_方案终稿.md`；
- 找不到源文件时立即失败；
- 支持标题、表格、列表、加粗、图片、链接、脚注等常用 Markdown；
- 使用 `stylepacks/` 中的参考样式；
- 生成封面、目录、页眉页脚和页码；
- 对表格重复表头并防止核心内容静默丢失；
- 将输出写入 `output/`，不覆盖历史版本。

构建依赖 Pandoc 与 python-docx。不可用时明确报错，不切换到硬编码示例方案。

### 阶段 9：渲染与视觉验收

每个最终 Word 必须渲染为页面图片检查：

```bash
python scripts/render_review.py <项目目录>/output/xxx.docx --output-dir <项目目录>/qa/render
```

检查全部页面：

- 封面、目录和正文分页是否正确；
- 标题是否落在页尾；
- 表格是否越界、断裂、缺列；
- 图片是否变形、模糊或遮挡；
- 页眉页脚是否出现在不应出现的位置；
- 字体、字号、行距、页码是否统一；
- 是否存在空白目录、空白页、占位符和未替换标记。

视觉验收不通过，修复后重新生成并重新渲染。

### 阶段 10：反馈迭代

收到带批注、修订、颜色标记或高亮的 Word 时：

1. 使用 `scripts/extract_feedback.py` 提取批注、修订和视觉标记；
2. 形成修改意见台账；
3. 回到 Markdown 和数据源修改；
4. 重新运行校验、评审、构建和渲染；
5. 使用 `scripts/compare_versions.py` 生成文字差异摘要。

禁止把直接字符串替换 Word XML 作为标准修改方式。OOXML 只用于明确、可测试且无法通过上层工具完成的修复。

## 6. 交付门槛

| 指标 | 要求 |
|---|---:|
| 用户强制需求覆盖率 | 100% |
| 高风险事实证据覆盖率 | 100% |
| 普通外部事实证据覆盖率 | ≥95% |
| 未关闭高优先级待确认事项 | 0 |
| 预算计算错误 | 0 |
| 正文与数据表关键数字冲突 | 0 |
| 未核验赛事具体名称 | 0 |
| 未经证实的首创/领先表述 | 0 |
| 课程与设备/空间断链 | 0 |
| 一票否决问题 | 0 |
| 明显页面排版缺陷 | 0 |

## 7. 交付文件

最终至少保留：

- `07_方案终稿.md`；
- `06_评审汇总.md`；
- `qa/validation_report.json`；
- `output/项目名称_vX.Y.docx`；
- `08_交付说明.md`。

对外只交付用户要求的文件；证据台账、评审报告和 QA 文件默认作为内部工作底稿，用户要求时再提供。
