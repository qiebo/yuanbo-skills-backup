# QA 清单

每次交付前至少完成以下检查。

## 数据一致性

- Task 1-8 的 checklist 已展示并逐项更新，未跳步。
- Task 3 使用 WorkBuddy/Codex 稳定登录策略：先运行一次 `npm run login:check`，失败后请求用户协助。
- 不得循环尝试自动登录，不得反复打开/关闭浏览器，不得重复提交凭证。
- 自动登录成功或用户确认手动登录完成前，不得进入网站取数。
- 网站登录成功后才进入网站取数；登录失败或后台学校身份无法确认时不得继续。
- `output/login_check.json` 不得包含账号、密码、cookie、token、authorization header 或浏览器 storageState。
- `validate_website_data.py` 已返回 `status: ok`；否则不得生成报告。
- Excel 目标学校名与网站后台学校名一致；如不一致，暂停并询问用户。
- 数据周期来自 Excel 月度行，不从报告模板文字推断。
- 学生账号总数来自网站学生管理分页总数。
- 累计交互总量按固定口径求和，不包含任何使用人数列。
- 本学期活跃为月度 `总使用人数` 最大值。
- 趋势表使用网站后台记录数趋势，报告中不得写作使用次数趋势。
- 漂流瓶覆盖学生如为发瓶学生去重，报告中必须标注该口径。
- AI 心语伙伴多口径不得混写。

## 报告内容

- 不出现 `{{placeholder}}`、`TODO`、`xxxx`、`###` 等调试文本。
- 高频问题进入正式报告前要过滤不适合公开呈现的敏感或不雅内容。
- 学校行动建议必须保留，不能只列数据。
- 结论要能从登记表追溯，不得用未获取字段补结论。
- 同一表格下方的分析段落不得机械复用相同句式或相同后半句。即使两个年级数据接近，也必须从年级场景、风险层级、管理动作或后续跟进方式上写出差异。
- 心理困扰因子和心理状态分级尤其要避免“同因子同解释”“同风险比例同解释”的套话，应按高一适应、高二分化、高三升学压力等实际场景区分表达。
- AI 模拟面试章节不得写未取证或系统未开放的申请场景，例如“港澳高校申请”；如果登记表没有年级分布，也不得写“用户以高三为主”等用户结构判断。
- 正式报告面向学校领导汇报，正文不得出现内部审计式限制说明或低信息描述，例如“当前已获取数据只能说明……不能推断……”“用于反复演练与复盘”“提供全真模拟……”这类句子。必要的口径限制写入登记表或备注，正文要改写为管理判断、价值判断和下一步动作。

## 文件质量

- Word 使用 `微软雅黑`，表头深蓝底白色加粗，表格内容垂直居中，关键数据分色加粗。
- Word 必须严格沿用模板结构，不能自行增删章节、模块或表格；默认模板应为 14 张表，且表格行列结构要与模板一致。
- Word 表格应尽量保持在同一页内；不得因为跨页重复表头而让一张逻辑表看起来像两张表。若版面空间不足，应优先通过分页前置、行不跨页断行、取消重复表头等方式保证阅读连续性。
- Word 必须是长报告形态，不能只输出表格摘要；建议不少于 60 段，并保留模块分析、价值判断和行动建议。
- 分析段落不能复述表格数值，必须说明这些数据对学校管理、年级运营、心理支持或升学服务有什么意义。
- 数据登记表至少包含：说明、核心指标、模块指标、网站记录趋势、内容与年级统计、模板字段路由、QA 与未决项。

## 推荐命令

```bash
python scripts/qa_report_bundle.py \
  --registry output/学校-生涯翼站报告数据汇总.json \
  --docx output/学校-生涯翼站使用情况学期分析报告.docx
```

如目标 agent 有 OfficeCLI，再运行：

```bash
officecli validate assets/templates/yizhan-report-template.docx
officecli view output/学校-生涯翼站使用情况学期分析报告.docx stats
```
