# 教育方案设计

## 概述

本skill提供从需求理解到Word交付的完整教育项目方案设计工作流。

核心能力：
1. 教育政策与学校背景调研
2. 方案顶层设计与章节大纲构建（概念词提炼＋结构隐喻＋学校基因融合，方法论见 `references/top_design_playbook.md`）
3. 专业级方案内容撰写（去AI化，正例库见 `references/writing_guide.md` 第九节）
4. **三方评审门控**（课程专家、商业专家、政策专家）
5. 格式化Word文档自动生成
6. 已有文档的迭代修改（老师/客户反馈驱动）

## 触发条件

在以下场景自动加载本skill：
- 用户要求编写/修改教育类项目方案、申报书、建设方案
- 用户提到"写方案"、"做申报"、"项目书"、"预算方案"
- 用户需要生成Word格式的学校/教育机构项目文档
- 用户对已有方案进行迭代修改（如去AI味儿、删减内容、调整结构）
- 用户要求根据老师/客户标注意见修改文档
- 用户需要制作配置清单、预算表、课程规划等方案配套文档

## 用户偏好（从实战中提炼）

> 以下偏好来自多个项目的真实交互反馈，务必严格遵守。

### 内容品质
- **反对虚构**：产品名称、价格、链接必须真实可验证，绝不编造。如无法确认，标注"待确认"并提醒用户
- **顶层设计须新颖、高端且贴合学校**：概念词要有记忆点，结构隐喻要自洽，必须钩住该校独有基因——须综合校训精神、校史文脉、建设优势、未来发展方向四维因子提炼，并以深入调研为前提，给出真实可验证的新颖点。模板化、换校名即用的设计一律不通过
- **竞赛信息必须校验**：写方案前须核实竞赛是否仍在举办（如NOC已停办、部分白名单赛事已调整），不可照搬旧模板中的赛事清单
- **数据有出处**：预算数字、课时安排、设备参数等必须有来源（供应商报价单/官网/政策文件），不拍脑袋
- **用户做事谨慎细致**，对交付文档要求准确真实

### 工作方式
- **先给计划再执行**：用户习惯先看完整计划再动手，不要直接开写
- **信息整理偏好汇总行**：同类信息（如多校配置清单）用汇总表呈现，不要逐行罗列明细
- **版本管理**：每次修改生成新版本（v1→v2→v3），保留历史版本，便于回溯
- **保留原始链接**：调研过程中发现的政策文件、产品信息链接保存到调研报告供后续查证

### 行业惯例
- **供应商报价须有依据**：所有设备价格必须基于供应商提供的实际报价单或官网价格，不得拍脑袋定价；如有特定供应商折扣规则，在需求确认阶段记录并遵守
- **预算申报方案**：描述建设内容与方向即可，不展开具体技术实现细节
- **竞赛/成果类内容**：概述参与方向和预期层级，不罗列具体赛项名称（除非用户要求）
- **学校基因必须融入**：方案核心理念须综合目标学校的校训精神、校史文脉、建设优势与未来发展方向深度结合（先调研后设计，禁止套模板、禁止只贴校训）

## 工作流

### 阶段一：需求确认与信息收集

1. 确认方案类型（预算申报 / 项目建设 / 课程建设 / 综合申报）
2. 确认目标学校基本信息（学校名称、学段、特色、现有基础、校训校风、**校史文脉、建设优势、未来发展方向**——四维新基因须逐项收集，禁止凭印象）
3. 确认建设方向与技术/课程主题
4. 确认预算级别与资金来源
5. 确认交付格式要求（Word页数、排版风格、是否需要目录）
6. **确认竞赛与升学出口**：学校期望对接哪些竞赛？强基计划？是否需要专利产出？
7. **确认高校衔接**：是否需要对接特定高校？高校名称和对接方式？

如用户未主动提供，需逐项询问，不要假设。

> **文档输出**：将收集到的需求信息整理为 `0_需求确认.md`，结构如下：
> - 项目基本信息
> - 核心需求摘录
> - 待确认事项与决策点
>
> 文件存放于当前工作目录，后续所有阶段文件均以阶段编号命名，便于追溯和迭代。

### 阶段二：政策与背景调研

根据方案主题进行必要的背景调研：
- 国家/地方教育政策（如科学教育加法、新课标要求、人工智能教育政策）
- 学校所在区域的教育特色与竞争格局
- 同类学校案例与建设标杆
- **竞赛有效性校验**：核实方案中拟提及的竞赛是否仍在举办，是否仍在教育部白名单中
- **高校合作可行性**：如需对接高校，调研该校是否有相关方向的实验室/导师资源

调研结果需简洁呈现，为方案提供政策依据和差异化定位。

> **资料组织要求**：所有写入方案的内容必须源自用户提供的资料或已校验信息。建立"资料类型→落位章节"映射（政策文件→背景、报价单→建设内容与预算、校史/校训/会议纪要/发展规划→基因钩子与缺口），每条结论标注来源链接。资料缺失须追问或标"待确认"，严禁虚构。详见 `references/material_organization.md`。

> **文档输出**：将调研成果写入 `1_调研报告.md`，结构如下：
> - 政策依据（文件名+核心要点+原文链接）
> - 背景分析（区域现状、差异化机会）
> - 参考案例（标杆学校做法）
> - 竞赛校验（方案拟涉及的竞赛清单+是否仍在举办）
> - 调研结论（对本项目的启示）
>
> 后续撰写时直接引用本文档内容，不需要再回头搜索。如需调整观点，在本文档中修改。

### 阶段三：大纲设计

基于需求与调研，设计方案章节结构。典型的教育项目方案结构：

```
第一章 项目背景与意义
第二章 建设目标与主要内容
第三章 建设方案（技术路线/空间布局/设备配置）
第四章 课程与教学体系建设
第五章 经费预算与资金来源
第六章 预期成效与保障措施
```

大纲设计原则：
- 申报类方案控制在6章以内，避免过于冗长
- 预算申报方案侧重"建设内容与方向"，技术细节点到为止
- 每章先给标题，再简述该章核心要点，经用户确认后再进入撰写
- **学校基因融入**：在核心理念部分预留校训精神、校史文脉、建设优势、未来方向的融合位置（详见 playbook 第五节的基因卡与融合五步）

#### 顶层设计提炼（金标准核心步骤，不可跳过）

大纲确定前，必须先完成顶层设计的提炼——这是方案"新颖、高端、贴合学校"的关键，也是本技能相对通用写作最大的差异点。按 `references/top_design_playbook.md` 执行：

1. **套用公式**：`[记忆点概念词] ＋ [结构隐喻] ＋ [学校基因钩子（校训精神＋校史文脉＋建设优势＋未来方向）] ＋ [新颖/首创点] ＋ [分层/分期逻辑]`。
2. **提炼概念词**：优先用"学校基因综合提炼法"——先完成四维调研（校训精神、校史文脉、建设优势、未来方向），从四维取交集、定主线，再配结构框架（一核两翼/三阶递进/闭环法等）。概念词 ≤8 字、有画面、换校名不能复用。
3. **锁定基因钩子**：基于阶段一建立的"学校基因卡"，从四维因子中找与项目主题的交集，把抽象基因转译为项目中的具体行为（校训转译为可观察行为、校史转译为古今场景、优势转译为可落地支撑、方向转译为建设主轴）。
4. **给出新颖/首创点**：须真实可验证（如"全国首个高中阶段XX实验室"）；无法确认则改用"对标/达到XX水平"。
5. **选章节范式**：先判定文类，再套对应骨架（详见 `references/top_design_playbook.md` 第七节「多家族 typology」）。金标准分五大家族：
   - **A 类 自由建设方案**（科创中心/实验室/工程教育/农场/美育融合/初中工程）：背景→顶层设计→建设内容→课程→预算→成效。
   - **B 类 标准申报书**（政府模板）：字段固定，但须把概念词与基因注入"项目概况/亮点特色"段。
   - **C 类 预算配置型**：无概念词，按能力模块分类＋测量依据。
   - **D 类 采购/合作协议型**（航空航天等多为此形态）：设备清单+技术参数+课时资源，顶层设计弱。
   - **E 类 定位汇报逐字稿**（科学高中/拳头项目）：按PPT页序、战略判断口吻，以确定性判断直接陈述定位（避免"不是…而是…"式AI对照结构）。
6. **自检**：用 playbook 第八节清单逐项核对，任一未达标重写，直至全部通过。

> 顶层设计确定后，将其概念词与结构隐喻贯穿背景、理念、内容三处，并在大纲的"核心理念/顶层设计"章明确标注。

> **文档输出**：将大纲写入 `2_方案大纲.md`，结构如下：
> - 章节标题（逐级展开）
> - 每章核心内容要点（用自然段落而非条目罗列）
> - 需要重点突出的方向
>
> 大纲确认后，在本文档上直接编辑内容，即为初稿撰写。迭代修改时，始终以大纲文档为工作底稿，调整后再生成Word。

### 阶段四：内容撰写（去AI化）

在 `2_方案大纲.md` 中直接填充各章节正文。严格遵守以下风格规范：

撰写方案正文时严格遵守以下风格规范：

**禁止使用的表达形式**：
- 条目罗列式（如"一是...二是...三是..."、"首先...其次...最后..."）
- 过度结构化的小标题套娃
- "综上所述"、"值得注意的是"、"不难发现"等AI高频过渡语
- 排比句堆砌
- 空洞的形容词罗列

**推荐的表达形式**：
- 自然段落叙述，段落之间用逻辑推进而非机械连接
- 专业术语准确使用，不刻意解释常识性概念
- 用具体数据、案例、政策文件支撑论点
- 适当使用长句与复合句，体现专家写作的层次感
- 语气客观、平实，避免推销感

**内容深度控制**：
- 预算申报方案：描述建设内容与方向即可，不展开具体技术实现细节
- 竞赛/成果类内容：概述参与方向和预期层级，不罗列具体赛项名称
- 设备配置：给出功能定位与数量，不写详细参数说明书

**严谨性与资料溯源**：
- 严格按 `references/material_organization.md` 组织内容：政策引文件名、设备引报价单、预算据实汇总、学校基因仅用本校核实资料（校训原文、校史事实、真实优势、官方规划）。
- 每条事实可回溯到 `1_调研报告.md` 的来源标注；缺失信息标"（待用户确认）"，不补虚构。

**语言风格（去AI化且正式规范）**：
- 遵循 `references/writing_guide.md` 禁用清单，并模仿其**第九节金标准正例库**的真实句式（背景四层一气呵成、概念词＋闭环、具象隐喻、校训转译为行为、无标签化收尾）。
- **校训/基因三处呼应**：在背景、理念、意义与价值三处反复回扣学校基因，而非文末贴一句"呼应校训"。
- 大量使用该校专属词汇（校训词、已有课程命名、地缘产业词），是消除"换校名即可复用"AI感的最有效手段。

**竞赛信息红线**：
- 每个提及的竞赛必须经过 `1_调研报告.md` 中的校验
- 已停办赛事（如NOC）严禁写入方案
- 如不确定赛事是否仍在举办，使用"全国学生信息素养提升实践活动等教育部白名单赛事"等泛指表述

### 阶段五：三方评审（关键门控）⚠️

> **这是交付前的必经关卡。未经三方评审一致通过的方案，不得提交给用户。**

方案初稿完成后，启动三方评审。使用Agent工具创建评审团队，三位评审员并行工作：

#### 评审员角色与评审维度

| 评审员 | 核心维度 | 关注要点 |
|--------|---------|---------|
| **课程评审员** | 课程体系与教学逻辑 | 课程分层是否合理、课时是否充足、竞赛出口是否可达、设备与课程是否匹配、学生能力进阶路径是否连贯 |
| **商业评审员** | 商业可行性与品牌安全 | 预算是否合理、设备配置是否冗余、品牌暴露风险、报价规则是否正确、竞品信息是否泄露 |
| **政策评审员** | 政策合规与实施风险 | 政策引用是否准确、时间节点是否可行、师资培训量是否充足、预算是否偏低、风险是否可控 |

#### 评审流程

```
方案初稿 → 三方并行评审 → 汇总评审报告 → 判断是否通过
                                              ↓
                                    全部通过 → 进入阶段六
                                    任一不通过 → 修改后重新评审
```

#### 评审标准

- **通过条件**：三位评审员均评分≥7.0分，且无P0级问题
- **P0问题**（必须修复才能通过）：事实性错误、虚构数据、政策引用错误、竞赛信息错误、品牌安全风险
- **P1问题**（强烈建议修复）：逻辑不连贯、课时不足、预算偏低、培训量不足
- **P2问题**（可选优化）：措辞优化、格式调整、补充附录

#### 评审文档输出

> - `评审报告_课程评审员.md`
> - `评审报告_商业评审员.md`
> - `评审报告_政策评审员.md`
> - `三方评审汇总.md`（含三位评审员评分、问题清单、改进建议、通过状态）
>
> **如评审未通过**：根据评审意见修改方案，修改后重新发起评审，直到三方一致通过。

### 阶段六：Word文档生成

使用 `scripts/generate_proposal.js` 脚本生成格式化Word文档。

前置检查：
1. 确认Node.js已安装
2. 确认全局安装docx包：`npm install -g docx`
3. 设置NODE_PATH指向全局node_modules（Windows: `$env:NODE_PATH='C:/Users/<user>/AppData/Roaming/npm/node_modules'`）

脚本使用方法：
1. 复制 `scripts/generate_proposal.js` 到工作目录
2. 在脚本头部修改文档元数据（标题、学校名、日期等）
3. 在脚本主体中按章节填充内容（使用提供的helper函数）
4. 执行 `node generate_proposal.js`

**关键编码规范**：
- 脚本中所有中文字符串必须使用Unicode转义（如 `\u5B8B\u4F53` 代替"宋体"）
- 中文引号使用 `\u201c` / `\u201d`（避免与JS字符串引号冲突）
- 破折号使用 `\u2014` 转义
- 单引号使用 `\u2018` / `\u2019`

> **文档输出**：生成最终Word文档的同时，将最终版Markdown内容写入 `3_方案终稿.md`。该文件作为"源文件"，后续任何修改都在此文件上进行，修改后重新生成Word即可。
>
> 文件命名约定：
> | 阶段 | 文件名 | 用途 |
> |-----|--------|------|
> | 需求确认 | `0_需求确认.md` | 记录需求与决策点 |
> | 调研报告 | `1_调研报告.md` | 政策依据、背景分析、参考案例 |
> | 方案大纲 | `2_方案大纲.md` | 章节结构+核心要点（后续直接在此填充正文） |
> | 方案终稿 | `3_方案终稿.md` | 完整正文Markdown源文件 |
> | 三方评审 | `三方评审汇总.md` + 3份评审报告 | 评审门控产物 |
> | Word输出 | `XX学校XX实验室建设方案.docx` | 最终交付物 |

### 阶段七：交付与迭代

交付时：
1. 生成docx文件并验证（用python-docx读取前30段确认结构）
2. 向用户说明文档结构与关键修改点
3. 询问是否需要调整

**迭代修改的工作方式**：
- 用户提出修改意见后，直接在 `3_方案终稿.md` 中编辑对应段落
- 修改完成后，重新运行 `generate_proposal.js` 生成新版Word
- 如需调整结构（加/删章节），先在 `2_方案大纲.md` 中修改，再同步到 `3_方案终稿.md`
- 如需补充调研内容，先在 `1_调研报告.md` 中更新，再引用到终稿
- **每次修改生成新版本**（v1→v2→v3），保留历史版本

常见迭代需求处理：
- "去AI味儿"：检查并替换所有条目罗列，改为段落叙述（修改 `3_方案终稿.md`）
- "太细了"：删除过度技术化的子节，保留方向性描述（修改 `3_方案终稿.md`）
- "加一节/删一节"：先在 `2_方案大纲.md` 调整结构，再同步到 `3_方案终稿.md`
- "调整措辞"：修改 `3_方案终稿.md` 对应段落后重新生成Word

### 阶段八：老师/客户反馈驱动修改（特殊流程）

> 当用户提交已标注修改意见的Word文档时（通常以红色文本标注），走此特殊流程。

#### 8.1 提取修改意见

1. 解包docx文件（docx本质为ZIP压缩的XML集合）
2. 搜索所有 `<w:color w:val="FF0000"/>` 标识的红色文本
3. 结合上下文推断每条红色标注所属章节和修改意图
4. 整理为编号清单，先向用户确认理解是否正确

#### 8.2 逐条修改

**Word XML修改规范**（重要！）：
- 文本可能跨多个 `<w:r>` 元素（因字体/颜色不同被拆分），需扩大匹配范围
- 替换时**必须保持XML标签闭合完整**，特别是 `</w:r>` 和 `</w:p>`
- 使用Python脚本读取文件，用 `repr()` 输出精确XML片段，再进行匹配替换
- 修改后必须用pack脚本重新打包，并验证XML格式正确

**修改操作流程**：
1. 复制当前版本为新版本文件（如 `（老师意见-已修改v2）.docx`）
2. 解包新版本文件
3. 在 `document.xml` 中定位并修改对应XML片段
4. 重新打包为docx
5. 验证修改结果（用Python读取docx确认文本正确、NOC等已删除内容确实不存在）

**常见错误与修复**：
| 错误 | 原因 | 修复 |
|-----|------|------|
| Edit工具匹配失败 | 字符串不完全一致 | 用Python `repr()` 输出精确片段 |
| pack报错 `mismatched tag` | 替换时误删 `</w:r>` 闭合标签 | 检查替换内容，补充闭合标签 |
| 中文乱码 | XML编码问题 | 确保UTF-8编码读写 |

#### 8.3 删除内容的连锁修改

当删除某个课程/课题时，必须同步处理：
- **正文引用**：总述中的数量（如"三个课题"→"两个课题"）
- **配套表格**：该课程对应的课程信息表（适用对象/编程语言/课时/赛事）
- **其他章节引用**：竞赛体系、预算表等处的关联内容
- **赛事清单**：如涉及已停办赛事，一并清理

### 配套文档制作流程

#### 配置清单

1. 收集供应商报价单（xlsx格式），提取设备名称、型号、单价
2. 按课程类别分组（A类/B类/C类 + 竞赛展示 + 服务 + 空间建设）
3. 使用建设项目清单模板（如有），保留模板格式和图片
4. 计算各类别小计和总计

2. 制作Deck Spec（逐页设计规范）

4. 嵌入场景图片（AI生成素材 + 供应商提取素材）
5. **PPT内容须与方案终稿一致**，竞赛信息同样需校验

## 最佳实践

1. **先给大纲再写全文**：永远先输出章节结构供用户确认，确认后再生成完整文档
2. **每阶段产出本地文档**：每个阶段完成后立即写入Markdown文件，不要只在对话中展示。这是后续迭代的工作底稿
3. **保留源链接**：调研过程中发现的政策文件、产品信息链接保存到 `1_调研报告.md` 供后续查证
4. **控制篇幅**：预算申报类方案正文控制在8000-15000字，过多内容反而稀释重点
5. **表格优于文字**：预算、设备配置、进度安排等用表格呈现，正文不做逐项朗读
6. **一次生成一版**：每轮修改重新生成完整文档，避免增量编辑引入格式错误
7. **Markdown为源，Word为交付物**：始终以 `3_方案终稿.md` 为内容源文件，Word仅作为格式化输出。修改时先改Markdown，再重新生成Word
8. **三方评审是硬门控**：方案终稿、配置清单等关键产物必须经过课程/商业/政策三方评审一致通过后才能提交给用户，绝不能跳过
9. **竞赛信息务必校验**：写方案前核实竞赛是否仍在举办，已停办赛事严禁写入
10. **删除内容要连锁检查**：删除某个课程/课题时，必须同步修改总述数量、配套表格、其他章节引用
11. **学校基因融入不是口号贴标**：核心理念须综合校训精神、校史文脉、建设优势、未来发展方向深度结合，而非简单在文末加一句"呼应校训"；基因提炼须以阶段一/二的四维调研为依据，禁止凭空贴标
